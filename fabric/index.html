02 Live Infrastructure · What's Actually Running
NginxLive
10 Subdomains · Multi-Site · Certbot SSL

Nginx managing ten separate virtual hosts on a single VPS. Each subdomain has its own server block, SSL certificate via Let's Encrypt Certbot, HSTS headers, and server_tokens off. Static sites serve directly from web root. Node.js applications proxy through internal ports. Rate limiting applied at the nginx layer for all public form endpoints.

PM23 Processes
BookForge · Conduit Hub · VA Contact Proxy

Three Node.js processes managed by PM2 with startup persistence. BookForge runs on port 3000 — the full writing management application. Conduit Hub runs on port 3100 — the automation dashboard querying the bookforge PostgreSQL database directly. VA Contact Proxy runs on port 3099 — handling all Telegram form submissions server-side so credentials never touch the client.

PostgreSQL16.13
bookforge Database · forge_master User

PostgreSQL 16 running locally. Single database — bookforge — with tables for users, books, entries, and conduit_log. The users table handles both Google OAuth identities and manual credential accounts with role-based access. The books table drives both the Book Forge application and the Press static catalogue rebuild. The conduit_log table records every automation trigger fired through the Conduit Hub dashboard.

03 Constellation Architecture · All Sites Mapped
Overview2026
One VPS. Ten Subdomains. One Coherent System.

The VectraArch constellation is not a collection of independent sites — it is a single engineered system running on one Ubuntu 24 LTS VPS. Nginx routes all traffic. PM2 keeps three Node.js processes alive permanently. A single PostgreSQL database serves both the writing application and the automation layer. Every subdomain shares the same stylesheet, the same SSL infrastructure, and the same philosophy: build it properly or don't build it at all.

Static7 Sites
Static HTML · Served Direct from Web Root

Seven sites served as static HTML directly from Nginx — no runtime, no process, zero overhead. All share /shared/va.css for a unified design system. Each has its own Nginx server block and Certbot SSL certificate.

A single PostgreSQL 16 instance serves both Book Forge and Conduit, with role-based access enforced at the application layer.

04 Security Layer · What's Hardened
01 · Transport
SSL / HSTS
Let's Encrypt · Certbot · Auto-renew

Every subdomain has a valid SSL certificate managed by Certbot with automatic renewal. HSTS headers enforce HTTPS with one-year max-age and includeSubDomains on the main domain.

02 · Tokens
Server-Side Credentials
Node.js Proxy · Never Client-Side

All API tokens and credentials — Telegram bot, DB passwords, session secrets — are held server-side only. A Node.js proxy on port 3099 handles all external API calls. No secrets in any HTML source.

03 · Rate Limiting
Nginx Rate Zones
contact zone · global zone · burst control

Two rate limit zones defined at the nginx http level. Contact endpoints limited to 5 requests per minute per IP with burst of 3. Global zone applied to the Book Forge application and manager interface.

04 · Auth
2FA · Google OAuth
TOTP via otplib · Passport.js

Book Forge supports TOTP-based two-factor authentication for manual accounts and Google OAuth for federated identity. Session-based auth with secure, HttpOnly, SameSite cookies. Role-based access control enforced at every route.

05 · Headers
Security Headers
X-Frame-Options · X-Content-Type · Referrer

X-Frame-Options SAMEORIGIN, X-Content-Type-Options nosniff, Referrer-Policy strict-origin-when-cross-origin, and server_tokens off applied across all nginx server blocks. No version disclosure.

06 · OS
Ubuntu Hardening
Ubuntu 24 LTS · Ansible-managed

Ubuntu 24 LTS with Ansible-managed configuration. Automated OS and package updates. Minimal attack surface — only necessary services running. Firewall rules managed at the VPS level with nginx as the only public-facing listener.
